## 1. 文档定位

本文档是 TurboFlow 足球 CLOB 预测市场的**分阶段交付路线图**。以 CLOB PRD v1（`TurboFlow足球CLOB产品需求文档_v1.md`）定义的 7 Tab、2 种市场类型为基础，先按**玩法逻辑族**给所有盘口分类，再按族的复杂度和数据依赖递进排入 4 个交付阶段。

**本文档不替代 PRD**——市场的详细定义（数据结构、交互行为、订单簿规范、交易系统规范等）仍以 PRD v1 为准。本文档只回答：**什么时候做什么、为什么这样排**。

### 1.1 CLOB 模式对交付路线的影响

与传统足球盘口的路线图相比，CLOB 模式引入以下新的基础设施依赖：

| 传统模式 | CLOB 模式 | 路线图影响 |
|---------|----------|-----------|
| 庄家定赔率 | 用户间订单簿撮合 | Phase 1 必须建设撮合引擎 |
| 即时投注 | 市价 + 限价单 | Phase 1 必须实现双订单类型 |
| 单向下注 | Yes/No 双向交易 | Phase 1 起所有盘口均需双向交易 |
| 下注即锁定 | 持仓可平仓、挂单可撤 | Phase 1 必须实现完整仓位管理 |
| 欧洲赔率 | ¢ 价格 + 归一化 | 数据转换管道从 Phase 0 起即完成 |
| 6 种 UI 类型 | 2 种市场类型 | 前端复杂度下降，后端撮合复杂度上升 |

## 3. 玩法逻辑族分类

CLOB PRD v1 的所有市场虽然只有 BinaryMarket 和 NegRiskEvent 两种 CLOB 类型，但其来源盘口按核心判定逻辑可归纳为 **11 个玩法族**。同一族内的盘口共享相同的判定机制，前端使用相同的数据转换路径。

每个族额外标注其在 CLOB 转换管道中的状态：
- **✅ 已转换**：当前 converter.ts 已实现转换逻辑
- **⚠️ 已过滤**：在 CLOB 模式中被有意排除（如平局返还、双胜彩）
- **🔄 已合并**：多个原始 binary 经 mergeDuplicateBinaries 合并为 NegRiskEvent

---

### 3.1 族 A：赛果族（N 选 1 赛果判定）

**核心机制**：常规时间结束后，判定比赛结果属于哪个选项。

| 盘口 | 来源 Tab | 转换来源 → CLOB 类型 | CLOB 状态 |
|------|---------|---------------------|----------|
| 胜平负 (1x2) | home | ButtonGroup(3) → NegRiskEvent | ✅ 已转换 |
| 平局返还 | — | — | ⚠️ 已过滤 |
| 双胜彩 | — | — | ⚠️ 已过滤 |
| 上半场 1x2 | halves | ButtonGroup(3) → NegRiskEvent | ✅ 已转换 |
| 下半场 1x2 | halves | ButtonGroup(3) → NegRiskEvent | ✅ 已转换 |

**过滤原因**：平局返还和双胜彩在 CLOB 模式中，用户可通过在「胜平负」市场同时持有多个 Outcome 的 Yes 来构造等效策略，无需独立市场。

---

### 3.2 族 B：大小球族（与盘口线比较）

**核心机制**：某个统计数值与预设盘口线比较，判定高于还是低于。

| 盘口 | 来源 Tab | 转换来源 → CLOB 类型 | CLOB 状态 |
|------|---------|---------------------|----------|
| 合计 O/U | home, goals, asian | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换（仅半球/四分之一球线） |
| 主队合计 / 客队合计 | goals | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 亚洲总进球 O/U | asian | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 上半场总计 / 下半场总数 | asian, halves | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 角球总数 O/U | corners | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 上半场角球总数 | corners | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 红黄牌总数 O/U | cards | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |

**CLOB 特殊处理**：所有 OddsTable 仅保留半球线（x.5）和四分之一球线（x.25/x.75），整数线被过滤（无法在二元市场中表达 push/退款）。

---

### 3.3 族 C：让分盘族（虚拟加减球后比较）

**核心机制**：给一方虚拟加或减若干球，再判定调整后的赛果。

| 盘口 | 来源 Tab | 转换来源 → CLOB 类型 | CLOB 状态 |
|------|---------|---------------------|----------|
| 亚洲让分盘 | home, asian | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 上半场让球 | halves, asian | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 下半场盘口 | halves, asian | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |
| 欧洲让球（3 列） | — | OddsTable(3列) | ⚠️ 已过滤 |
| 角球盘口 | corners | OddsTable(2列) → NegRiskEvent 或 Binary | ✅ 已转换 |

**过滤原因**：三列 OddsTable（欧洲让球）无法直接映射为二元或互斥市场。

---

### 3.4 族 D：是/否族（二元条件判定）

**核心机制**：某个条件成立 → Yes 赢，否则 → No 赢。在 CLOB 中天然对应 BinaryMarket。

| 盘口 | 来源 Tab | 转换来源 → CLOB 类型 | CLOB 状态 |
|------|---------|---------------------|----------|
| 两队都得分 (BTTS) | home, goals | ButtonGroup(2) → BinaryMarket | ✅ 已转换 |
| 上半场两队得分 | halves | ButtonGroup(2) → BinaryMarket | ✅ 已转换 |
| 总进球数单/双 | goals | ButtonGroup(2) → BinaryMarket | ✅ 已转换 |
| 其他是/否类盘口 | specials | ButtonGroup(2) → BinaryMarket | ✅ 已转换 |

---

### 3.5 族 E：精确命中族（猜中精确结果）

**核心机制**：预测的精确数值或区间与实际结果匹配则赢。在 CLOB 中均映射为 NegRiskEvent。

| 盘口 | 来源 Tab | 转换来源 → CLOB 类型 | CLOB 状态 |
|------|---------|---------------------|----------|
| 正确进球（全场 5x5） | home | ScoreGrid → NegRiskEvent | ✅ 已转换 |
| 上半场正确进球 (3x3) | home, halves | ScoreGrid → NegRiskEvent | ✅ 已转换 |
| 总进球数 | goals | RangeButtons → NegRiskEvent | ✅ 已转换 |
| 1x2 & 两队得分 | home | ComboGrid → NegRiskEvent | ✅ 已转换 |
| 半场/全场 | home | ComboGrid → NegRiskEvent | ✅ 已转换 |
| 其他 ComboGrid 组合 | goals | ComboGrid → NegRiskEvent | ✅ 已转换 |

---

### 3.6 族 F：组合族（两维交叉同时命中）

在传统模式中为独立的族，在 CLOB 中 ComboGrid 统一转换为 NegRiskEvent——每个交叉格变为一个 outcome。逻辑上仍属组合族，但 CLOB 类型层面与精确命中族无差异。

### 3.7 族 G：奇偶族（奇偶判定）

|------|---------|----------|----------|
| 总进球数单/双 | goals | BinaryMarket | ✅ 已转换 |
| 角球奇数/偶数 | corners | BinaryMarket | ✅ 已转换 |

### 3.8 族 H：先/后族（首个或末个事件归属）

| 盘口 | 来源 Tab | CLOB 类型 | CLOB 状态 |
|------|---------|----------|----------|
| 谁先进第 1st 球 | goals | NegRiskEvent(3) | ✅ 已转换 |
| 最后一个进球属于 | goals | NegRiskEvent(3) | ✅ 已转换 |
| 1st 角球 | corners | NegRiskEvent(3) | ✅ 已转换 |
| 最后角球 | corners | NegRiskEvent(3) | ✅ 已转换 |
| 1st 红黄牌 | cards | NegRiskEvent(3) | ✅ 已转换 |

---

### 3.9 族 I：进球者族（球员级进球判定）

**核心机制**：预测某位球员是否进球。在 CLOB 中，每个球员先生成一个 BinaryMarket，同 groupTitle 的多个 binary 经 mergeDuplicateBinaries 合并为单个 NegRiskEvent。

| 盘口 | 来源 Tab | CLOB 类型 | CLOB 状态 |
|------|---------|----------|----------|
| 任何时间进球队员 | specials（从 goalscorer 注入） | NegRiskEvent（🔄 已合并） | ✅ 已转换（前 10 名球员） |

**Phase 3 扩展方向**：1st 进球队员、最后进球队员、更多球员覆盖。

---

### 3.10 族 J：球员统计族（球员级数据达标判定）

当前 CLOB 模式不纳入。Phase 3 扩展方向：球员助攻、射门、射正、罚牌等达标市场。

---

### 3.11 族 K：时间区间族（事件发生时段判定）

当前 CLOB 模式不纳入。Phase 3 扩展方向：1st 进球时间段（10分钟/15分钟切片）。

---

### 3.12 能力层 Z：组合投注

传统模式的 SGM / Bet Builder 在 CLOB 模式中没有直接对应物——CLOB 用户通过同时在多个独立市场持仓来构建组合头寸，平台不提供串关赔率相乘。

Phase 3 可选方向：「组合头寸」功能，将多个市场的持仓打包显示盈亏，但不改变底层独立撮合机制。

## 8. 风险与注意事项

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| 撮合引擎性能 | CLOB 模式下每个 outcome 都有独立订单簿，NegRiskEvent 的多 outcome 市场会显著增加撮合负载 | 分 outcome 独立撮合线程；优先优化热门市场 |
| 流动性冷启动 | 新平台缺少做市商，订单簿深度不足，用户体验差 | 平台初期自营做市；引入自动做市商（AMM）作为流动性兜底 |
| NegRiskEvent 定价一致性 | 多个 outcome 的 yesPrice 总和可能偏离 100（套利空间） | 引入归一化约束或套利机器人自动平衡 |
| 数据源质量 | 第三方 feed 延迟或错误可能导致错误结算 | 多源校验 + 人工兜底 + 结算延迟窗口 |
| 半场结算时序 | 半场结束的精确时间判定依赖数据源 | 与数据方确认半场终了推送逻辑 |
| Live 滚球 Suspended 时序 | 进球后 suspend 太慢可能导致用户以过时价格成交 | 事件检测 → 立即 suspend → 价格调整 → 恢复，全链路 < 1s |
| 球员数据完整性 | 球员级事件（助攻、射正等）覆盖率可能不足 | Phase 3 前充分评估数据源覆盖率 |
| 监管合规 | 预测市场在不同司法管辖区的法律地位不同 | 按目标市场法律框架调整产品形态 |
