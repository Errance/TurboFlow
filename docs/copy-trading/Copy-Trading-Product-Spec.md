

## 一、产品概述

合约跟单允许**跟单者**选择**带单员**，按带单员的开仓订单自动跟单开平仓；带单员可从跟单者的盈利中获得**分润**。

**角色定义**

- **带单员**：申请通过后可在交易页通过专用带单路由发起带单；带单员开仓时，系统自动为其跟随者按各自配置开仓；带单员平仓时，关联跟单者仓位一并平仓。
- **跟单者**：在带单广场发现并选择带单员，完成跟单设置后由系统自动跟单开平仓；可查看当前跟单持仓、历史记录，并管理资金。

**账户体系**

- 跟单账户与合约账户**相互独立**，两边资金流水单独统计。
- 跟单者开始跟单时，资金从合约账户转入跟单账户；停止跟单后，剩余资金转回合约账户。
- 所有跟单仓位以 **USD 稳定币**计价结算。平台支持 USDC/USDT，分润计算统一按平台定义的稳定币记账单位处理，精度见 4.16 节。

**核心价值**

跟单者省心复制带单员策略；带单员通过分润获得额外收益。

## 二、带单员流程与规则

### 2.1 申请成为带单员

**系统前置校验**（任一不满足则申请失败）：

1. 账户下所有合约**无仓位、委托单、计划单**
2. 账户**未跟随他人订单**

两项校验均通过后进入**人工审核**：

- 审核通过 → 申请成功，账号获得带单员资格
- 审核拒绝 → 申请失败，需重新申请

> 具体审核标准（如收益率、交易笔数、盈利笔数等）由运营人工把控。

### 2.2 带单策略配置

每个带单员拥有**一套**带单策略，可配置以下参数：

| 配置项 | 说明 |
|--------|------|
| 最低跟单金额 | 带单员自行设置，跟单者资产不满足则无法跟随 |
| 分润比例 | 平台按带单员等级设置上限；带单员在 >0 且 ≤上限 的范围内自行选择比例（见第四章） |
| 锁定期 | 可选开关。开启后：跟单者在锁定期内不可停止跟单、不可转入或转出资金。关闭后按正常资金规则执行 |
| 公域/私域 | 公域：列表可见且信息公开；私域：列表可见但详情隐藏，跟单者需输入邀请码解锁 |

**展示规则**

- 公域带单员正常显示在带单广场，支持搜索；
- 私域带单员在广场列表中可见，但具体统计数据、交易记录等信息不可见，需邀请码解锁；
- 带单员账户的**合约总权益 ≥ 500 USDC** 时，系统考虑将其展示在列表；是否展示及排名由人工审核决定。

**私域邀请码管理**

- 带单员可自行生成邀请码，可生成**多个不同的码**，每个码可供**多人使用**
- 带单员可随时 **disable（禁用）** 已生成的码；禁用后该码不可再被新跟随者使用
- 已通过该码加入的跟随者**不受影响**，继续保留跟随关系

### 2.3 发起带单

带单员在交易页通过**专用带单路由**开仓，该路由下的订单即为带单仓位，系统自动触发跟单逻辑。通过普通交易路由开仓的订单为带单员自有仓位，**不触发跟单**。

- 同一合约同一方向可创建多笔带单订单。
- 带单员可在发起带单时**设置止盈止损**，展示止盈止损价供跟单者参考。

**带单仓位的平仓方式**

带单员平仓后，当前仓位关联的所有跟单者仓位一并平仓。支持以下方式：

- 市价平仓
- 市价全平
- 止盈止损触发平仓

平仓同时生成**明细订单**与**汇总订单**。

### 2.4 跟随者管理

**带单员不可单方面移除个别跟随者**，也不可单方面终止与个别跟单者的跟单关系。如需终止特定跟单关系，带单员可通过客诉、申诉或其他运营渠道发起申请，由客服和平台根据风控、合规、争议处理等规则人工处理。带单员如需停止带单，仅可通过停止整个带单组合或退出带单身份的方式终止全部相关跟单关系。

### 2.5 带单员退出

**停止带单（保留账号）**
- 停止接收新跟单
- 系统立即将所有跟单者的仓位**市价全平**
- 触发即时分润结算，按 HWM 机制处理（见 4.8.2 / 4.14.1 节）

**注销带单员身份**
- 停止带单，处理同上
- 账号退出带单员列表

> 带单员主动退出属于正常收口，不影响已发生的交易事实。退出触发即时结算，分润按 HWM 机制正常处理（见 4.14.1 节）。

### 2.6 平台封禁带单员

紧急处理流程（与主动退出不同）：

1. 立即**冻结**该带单员的所有带单功能，不再触发新跟单
2. 所有关联跟单者仓位**立即市价强制全平**
3. 全平产生的盈亏纳入已实现净利润，计算 accrued 和 reserved 后，将 reserved **冻结**（设置 `reserve_hold_reason = lead_banned`），阻止实际支付，待调查结果决定是否发放（详见 4.14.2 节）
4. 通过**站内信**通知所有跟单者封禁原因及后续补偿措施

> 封禁属于平台风险处置，与带单员主动退出（2.5 节）的核心区别：封禁场景下 reserved 被冻结暂不支付，需待调查结果决定是否发放；主动退出则按标准流程即时结算。

## 三、跟单者流程与规则

### 3.1 发现带单员

带单广场提供三个 Tab：

| Tab | 说明 |
|-----|------|
| **公域** | 所有通过审核的公开带单员，支持搜索与筛选 |
| **私域** | 私域带单员的列表入口，具体信息隐藏，需输入带单员提供的邀请码解锁后方可查看详情并跟单 |
| **收藏** | 跟单者收藏的带单员列表，纯展示，无特殊业务逻辑 |

### 3.2 开始跟单前的校验

系统依次校验（任一失败则无法跟随）：

1. 跟单者资产 ≥ 带单员设置的**最低跟单金额**
2. 带单员**近 30 日有开仓记录**（无记录时提示并阻止跟随）

### 3.3 跟单设置参数

跟单者在确认跟随前需完成以下设置：

**① 跟单类型（必选）**

| 类型 | 说明 |
|------|------|
| **Fixed Amount（固定金额）** | 每笔跟单订单使用固定 USDC 金额，适合控制单笔风险；转入额外资金不影响每笔跟单金额 |
| **Fixed Ratio（固定比例）** | 设置总投资金额，系统按比例自动跟随带单员每笔仓位大小；转入额外资金后，每笔跟单金额按新余额等比例增大 |

总投资金额范围：10 – 300,000 USDC（参考币安，具体上下限以运营配置为准）。

**② 现有持仓处理（必选）**

| 选项 | 说明 |
|------|------|
| 不复制现有持仓 | 仅跟随新开仓，带单员已有仓位不复制 |
| 复制更好入场 | 当前市价优于带单员入场价时才复制现有仓位 |
| 复制全部现有持仓 | 以市价复制带单员所有已有仓位 |

**③ 交易对白名单（可选）**

- 自定义跟随哪些交易对，未选中的交易对不跟单
- 可选择「自动加入新上市交易对」

**④ 保证金模式**

- 固定模式：全仓（Cross）或逐仓（Isolated）
- 跟随带单员：与带单员保持一致

**⑤ 杠杆倍数**

- 固定倍数：1x – ？x
- 跟随带单员：与带单员保持一致

**⑥ 最大入场滑点**

- 范围：0.1% – 3%
- 默认：BTC/ETH 为 0.5%，其他交易对为 1.5%
- 仅保护入场价，不保护平仓价

**⑦ 仓位级止盈止损（TP/SL）**

可选，开仓后系统自动为每个仓位挂 TP/SL 单。ROI% 为正数，方向由系统根据多空自动判断：

| 方向 | TP 触发价 | SL 触发价 |
|------|-----------|-----------|
| 多头（Long） | 入场价 × (1 + ROI% / 杠杆) | 入场价 × (1 − ROI% / 杠杆) |
| 空头（Short） | 入场价 × (1 − ROI% / 杠杆) | 入场价 × (1 + ROI% / 杠杆) |

> 注意：止损比例过小时，触发价可能与当前市价倒挂导致挂单失败。编辑跟单设置后的 TP/SL 仅对新仓位生效，已有仓位的 TP/SL 不会被自动更新或取消。

**⑧ 总止损（账户级）**

当跟单账户的保证金余额亏损达到设定值时，系统自动市价全平所有跟单仓位并停止跟单。支持两种设置方式：

**按 USDC 金额：**
- 设定一个止损触发余额绝对值
- 例：投入 100 USDC，设止损余额 40 USDC → 余额跌至 40 USDC 时触发

**按百分比：**
- 公式：止损触发值 = 当前余额 × (1 − 止损率%)
- 例：余额 100 USDC，设止损率 60%（即最多可接受亏损 60%）→ 触发线 = 100 × (1−60%) = 40 USDC
- 当余额跌至 40 USDC 时触发

**资金操作后的止损重算（与跟单类型无关）：**

每次转入资金后，百分比模式的止损触发值按**当前余额**重新计算：

- 新止损触发值 = 转入后新余额 × (1 − 止损率%)
- 例：余额 100 USDC，设 60% 止损率（触发线 40 USDC）→ 转入 60 USDC 后余额 160 USDC → 触发线重算为 160 × 40% = 64 USDC
- USDC 绝对值模式的止损触发值**不重算**，维持原设定的绝对值

**⑨ 锁定期（如带单员开启了锁定期）**

- 跟单者须确认并接受锁定期条款后方可跟随
- 锁定期内不可停止跟单、不可转入或转出资金

### 3.4 跟单执行逻辑

带单员开仓后，系统对每位跟随者依次执行：

1. **成本检查**：跟单账户资产是否充足？
   - 不足 → 本次跟单失败，**立即触发系统自动取消跟随**
2. **仓位上限分配**：带单金额总计（带单员 + 所有跟单者）是否超过平台该币种可开上限？
   - 未超出 → 各跟单者按实际设置金额开仓
   - 超出 → 以平台上限为准，带单员优先占用，**剩余额度按跟单者金额占比 pro-rata 分配**
   - 分配后若某跟单者的实际可开金额 < 最小开仓量，该笔跟单**跳过**（不触发成本不足的自动取消逻辑）
3. **开仓**：校验通过 → 开仓跟随成功

> **订单拆分与跟单触发规则（对齐币安）**：
> - **市价单**：以带滑点保护的 IOC 限价单模式成交，下单时**立即触发**跟单
> - **限价单**：等订单**完全成交**后触发跟单，跟单者以市价单执行
> - **多笔独立订单**：每笔订单**单独触发**跟单

带单员强平时，所有跟单者对应的跟单仓位同步触及平仓。

### 3.5 资金管理

| 操作 | 规则 |
|------|------|
| **转入（追加）** | 跟单进行中随时可追加资金；Fixed Ratio 模式下转入后每笔跟单金额等比例增大；百分比止损触发线按新余额重算 |
| **转出** | 跟单进行中**不允许转出**（全仓/逐仓均不可）；如需转出，须先停止跟单，仓位全平后资金自动转回 |
| **锁定期内** | 转入与转出均**不可操作** |

**全仓/逐仓转出说明：**

- **全仓（Cross）模式**：共享保证金，转出会直接降低所有仓位的维持保证金，可能触发强平，跟单期间不允许转出
- **逐仓（Isolated）模式**：各仓位保证金独立，但同样需先停止跟单才可转出

### 3.6 停止跟单

三种取消跟随方式：

| 方式 | 触发 | 已有仓位处理 |
|------|------|-------------|
| **主动取消** | 跟单者手动停止跟随 | **立即市价全平**所有跟单仓位 → 触发即时分润结算（见 4.8.2 节）→ 剩余资金转回 |
| **平台干预** | 平台/客服基于投诉、风控、合规等原因终止跟单关系 | 仓位全平 → 触发即时结算 → 按标准流程处理 |
| **系统取消** | 见下方触发条件 | 已有仓位继续跟随带单员平仓；跟单者可手动平仓 |

**系统自动取消跟随的触发条件：**

1. 跟单者**1 次**因**本金不足**跟单失败
2. 带单员**近 30 日无新开仓**

## 四、结算与分润

### 4.1 核心原则

1. **高水位线（HWM）**：带单员仅对跟单者的**净新增利润**收取分润。历史亏损必须先被完全补回，才能产生新的分润。
2. **Full MTM（对齐币安）**：计费基于「已实现净利润 + 未实现 PnL」。正向和负向未实现 PnL 均实时参与计费基准计算。
3. **关系级计算**：HWM 和分润金额按"跟单者 × 带单组合"维度独立维护，不跨组合、不跨关系共享。
4. **不自然重置**：HWM 不按自然周期（季度、年度）自动重置，仅在跟单关系终止并完成最终结算后重置。
5. **不预扣**：系统不在每次盈利平仓时直接从跟单者账户扣款，改为 reserve（待结算冻结）。
6. **三段金额状态模型**：分润金额分为 accrued（应计）/ reserved（待结算）/ settled（已结算）三个独立账务状态。应计层持续计算理论分润，现金层仅在满足支付条件时实际划转，中间通过 reserve 机制在触发点确认待结算义务。

### 4.2 术语定义

本章中以下术语含义统一，全文不使用其他近似表述替代：

| 术语 | 定义 |
|------|------|
| **账户总权益** | 跟单账户当前市值，含保证金、可用余额及未实现 PnL |
| **保证金占用** | 当前持仓所需的保证金总额 |
| **最低维持保证金** | 维持当前仓位不被强平所需的最低保证金 |
| **可分配余额** | 账户总权益 − 保证金占用 − 最低维持保证金。用于结算/支付/风控判断（reserve 上限、支付条件等） |
| **展示可用余额** | 账户总权益 − 保证金占用 − reserved_profit_share。用于前端展示跟单者当前可操作余额 |
| **linked positions** | 由该跟单关系复制出来、尚未完全平仓的仓位集合（详见 4.11 节） |
| **关系终止** | 跟单关系不可逆地结束（停止跟单、关闭组合、账户终止、平台/客服人工终止等），触发最终结算 |

### 4.3 净利润口径

#### 4.3.1 已实现净利润的定义

**已实现净利润**从交易记录层面直接聚合，不依赖账户余额推导：

```
已实现净利润 = Σ 所有已平仓复制仓位的盈亏
             − 交易手续费
             − 资金费（Funding Fee）
             − 保险基金/清算相关费用
```

#### 4.3.2 追加入金隔离

跟单期间允许追加资金（转入），但追加入金**不计入交易利润**。已实现净利润的计算与账户余额变动脱钩，确保充值不污染 HWM。

> 注：跟单期间不允许转出资金（见 3.5 节），因此无需处理提现对净利润的影响。

#### 4.3.3 费用扣除范围

以下费用在计算已实现净利润时**必须扣除**：

| 费用类型 | 说明 |
|---------|------|
| Trading Fee | 开仓/平仓交易手续费 |
| Funding Fee | 永续合约资金费率结算 |
| Insurance / Liquidation Fee | 强平清算相关费用 |
| ADL 损益 | 自动减仓（Auto-Deleveraging）导致的损益 |

**不扣除**：跟单者与带单员之间因滑点差异导致的入场价差异（copy slippage），该差异属于市场执行风险，由跟单者自行承担。

### 4.4 分润计费公式

#### 4.4.1 统一应计公式

```
应计分润基准 = 已实现净利润 + 未实现PnL

accrued_profit_share = Max(应计分润基准 × 分润比例 − settled_profit_share, 0)
```

- `已实现净利润`：见 4.3 节
- `未实现PnL`：当前持仓的浮动盈亏，正负均参与计算（Full MTM，对齐币安）
- `分润比例`：带单员自选比例（>0 且 ≤ 等级上限）
- `settled_profit_share`：已实际支付给带单员的累计金额

该公式天然包含 HWM 语义：当应计分润基准回撤导致 `基准 × 分润比例` 低于 `settled_profit_share` 时，`accrued` 归零，带单员无新增分润。

#### 4.4.2 公式说明

本公式全局统一，不因场景而切换。无论有无持仓，均使用同一公式。

当所有仓位已平仓（未实现PnL = 0）时，公式自动退化为纯已实现口径：

```
accrued_profit_share = Max(已实现净利润 × 分润比例 − settled_profit_share, 0)
```

#### 4.4.3 分润比例规则

- 平台按**带单员等级**设置分润比例上限（具体数值由运营配置）
- 带单员在 >0 且 ≤ 上限范围内自行选择分润比例
- 带单员升级后，**升级前已建立的跟单关系按原比例**，新建立的跟单关系按新比例

### 4.5 金额状态定义

#### 4.5.1 三态模型

分润金额分为三个独立的账务状态，不可混淆：

| 状态 | 字段名 | 含义 | 变动规则 |
|------|--------|------|---------|
| **应计** | `accrued_profit_share` | 实时理论应计分润。随已实现 PnL 和未实现 PnL（含浮盈浮亏）波动而变化，反映当前经济状态下的理论分润义务，**不代表**平台已确认的欠付 | 实时计算，可升可降 |
| **待结算** | `reserved_profit_share` | 平台在特定触发点确认的待结算分润义务。其值以最近一次有效结算检查时刻的结果为准，后续可在新触发点被重新确认、上调或下调 | 仅在特定触发点刷新（见 4.9 节），不实时抖动 |
| **已结算** | `settled_profit_share` | 已实际划转给带单员的分润金额 | 仅在支付完成后增加，不可回退 |

#### 4.5.2 关键约束

1. `reserved_profit_share` 不要求始终 ≤ 实时 `accrued_profit_share`；其值以最近一次有效结算检查/确认时刻的结果为准，可在后续触发点被重新确认、上调或下调
2. `reserved_profit_share ≤ Max(可分配余额, 0)`（reserve 不侵蚀维持保证金，在每次刷新时执行此检查）
3. `settled_profit_share` 单调递增，不可回退
4. `accrued_profit_share` 在 HWM 水位线以下时为 0

### 4.6 关键字段清单

每个"跟单者 × 带单组合"维护一份：

| 字段 | 类型 | 说明 |

### 4.7 结算状态机

#### 4.7.1 状态定义

| 状态 | 含义 |
|------|------|
| `normal` | 正常运行中，未到结算窗口或无应计分润 |
| `payable_pending` | 到达结算窗口或事件触发后，等待系统执行支付 |
| `settled` | 本次结算已完成支付 |
| `insufficient` | 满足支付条件但跟单者账户余额不足以覆盖 reserved 金额 |

#### 4.7.2 状态流转

```
                  ┌──────────┐
                  │  normal  │
                  └────┬─────┘
                       │ 结算窗口到达 / 事件触发
                       ▼
               ┌───────────────┐
               │payable_pending│
               └───────┬───────┘
                       │
                       ▼
            ┌─ hold_reason 非空? ─┐
            │ 是                  │ 否
            ▼                     ▼
     保持 payable_pending   ┌── 余额充足? ──┐
     （阻止支付，等待解冻）  │ 是             │ 否
                            ▼                ▼
                     ┌──────────┐   ┌──────────────┐
                     │ settled  │   │ insufficient │
                     └────┬─────┘   └──────┬───────┘
                          │                │
                          │ 重置           │ 余额恢复
                          ▼                ▼
                   ┌──────────┐    ┌───────────────┐
                   │  normal  │    │payable_pending│
                   └──────────┘    └───────────────┘
```

#### 4.7.3 `insufficient` 状态处理规则

当跟单者账户余额不足以覆盖 `reserved_profit_share` 时：

1. 标记 `settlement_status = insufficient`
2. 不执行支付，不减免应付金额
3. 在下一个结算窗口或触发事件时重新检查
4. 如果跟单关系终止（停止跟单/强平），从跟单者最终可分配余额中**尽可能支付**；不足部分进入未结清状态，后续处理方式由平台清结算与风控规则决定

> **为什么 reserved 在刷新时已受边界约束仍可能进入 insufficient？** `reserved_profit_share` 在刷新时受 `Max(可分配余额, 0)` 约束，但刷新后到实际支付前，跟单者的权益可能因市场波动、保证金变化、手续费/资金费扣减、或多个跟单关系并发结算争抢余额等原因而下降，导致支付时余额不足。

### 4.8 结算触发与节奏

#### 4.8.1 固定结算窗口

- **频率**：每周一次
- **时间**：每周一 00:00 UTC+8
- **行为**：系统遍历所有活跃跟单关系，按 Full MTM 公式重算 `accrued_profit_share` 并刷新 `reserved_profit_share`，逐一尝试结算支付
- **有仓位时**：无论有无 linked positions，均正常结算支付（Full MTM 已包含未实现 PnL，无需等待平仓）

#### 4.8.2 事件触发即时结算

以下事件发生时，系统立即对相关跟单关系尝试结算：

| 事件 | 触发说明 |
|------|---------|
| 跟单组合无仓位 | 该关系下所有 linked positions 全部平仓，且无挂单 |
| 跟单者主动停止跟单 | 主动取消，仓位立即全平后触发结算 |
| 系统自动取消跟随 | 仅停止新单复制，不强平旧仓；已有仓位继续跟随带单员平仓，待 linked positions 全部自然清零后由"跟单组合无仓位"事件触发结算 |
| 带单员关闭组合 | 停止带单或注销 |
| 跟单者提取利润 | 预留，MVP 阶段不可用（跟单期间不允许转出） |
| 强平 / 异常终止 | 跟单者触发账户级止损、带单员被强平、平台封禁等 |
| 平台/客服人工终止 | 平台基于投诉、风控、合规、争议等原因终止特定跟单关系。该场景属于平台干预，不属于带单员产品功能，也不构成带单员单方面终止权限 |

### 4.9 Reserve 刷新规则

#### 4.9.1 刷新时点

`reserved_profit_share` **仅在以下触发点**刷新，日常不跟随 accrued 实时波动：

1. 每周固定结算窗口
2. 跟单者主动停止跟单
3. 带单员关闭组合
4. 跟单者提取利润（预留）
5. 强平 / 异常终止
6. 平台/客服人工终止特定跟单关系

#### 4.9.2 刷新逻辑

刷新时，以触发时刻的 `accrued_profit_share` 为基准，受可分配边界约束：

```
reserved_profit_share = Min(
    触发时刻的 accrued_profit_share,
    Max(可分配余额, 0)
)
```

刷新后，`reserved_profit_share` 保持不变直到下一个触发点。期间 `accrued_profit_share` 可能因 PnL 波动而低于 `reserved`，这不构成异常——`reserved` 代表的是平台在确认时点的义务快照。

#### 4.9.3 对跟单者的影响

`reserved_profit_share` 是平台在特定触发点确认的待结算分润义务。在当前版本中（跟单期间不允许转出），它作为结算时必须优先划转的金额，实质上减少跟单者的展示可用余额；如未来开放跟单期间利润提取，也将进一步占用可提现额度。

- 跟单者端展示：账户余额正常显示全额，另起一行显示"待结算分润: XX USDC"
- 关系终止结算时，实际可划转金额受可分配余额（见 4.2 节）约束

#### 4.9.4 对带单员的影响

- 带单员端展示"累计已结算分润"、"当前待结算分润"、"当前冻结待结算分润"和"预估应计分润"
- 当前待结算分润 = 所有活跃关系中 `reserve_hold_reason = null` 的 `reserved_profit_share` 之和
- 当前冻结待结算分润 = 所有活跃关系中 `reserve_hold_reason ≠ null` 的 `reserved_profit_share` 之和
- 预估应计分润 = 所有活跃关系的 `accrued_profit_share` 之和（实时参考值）

### 4.10 实际支付条件

`reserved_profit_share` 转为 `settled_profit_share`（真正打款）**必须同时满足**：

1. `reserved_profit_share > 0`
2. `reserve_hold_reason = null`（未处于冻结状态）
3. 跟单者**可分配余额 ≥ reserved_profit_share**

> Full MTM 下无需等待平仓：公式已包含未实现 PnL，每周窗口到达时即可结算支付，无论是否有持仓。

支付完成后：
- `settled_profit_share += 本次支付金额`
- `reserved_profit_share -= 本次支付金额`
- 资金从跟单者账户划转至带单员现货钱包
- `settlement_status` 更新为 `settled`，随后重置为 `normal`

#### 4.10.1 并发结算顺序

当同一跟单者有多个活跃跟单关系需同时结算时，按 `relationship_id` 升序依次执行，每次支付前重新检查可分配余额，避免共享余额竞争导致不一致。

### 4.11 Linked Position 定义

**Linked position** = 由该跟单关系复制出来、尚未完全平仓的仓位集合。

- 部分减仓（Partial Close）**不算**关闭，只有该复制仓位 **fully closed** 才移出集合
- 带单员通过**普通交易路由**开的自有仓位**不属于** linked positions，不影响结算判断
- 判断"无 linked positions"时，同时检查是否有未成交的**挂单**（限价单、止盈止损单）

### 4.12 分润案例

以下案例假设分润比例 **10%**，所有仓位为 USDC 计价永续合约。

#### Case 1：HWM 基本运作 — 连续四周交易

> 假设各结算时点均已全平，未实现 PnL = 0，统一公式退化为纯已实现口径。

初始状态：`settled = 0`

| 周 | 当周已实现PnL | 累计已实现净利润 | accrued (基准×10% − settled) | reserved | settled | 说明 |
|---|-------------|----------------|------|------|------|------|
| 1 | +200 | +200 | 20 | 20 | 0 | 周一窗口，无仓位，可结算 |
| 结算 | — | — | 0 | 0 | 20 | 支付 20，settled 累计 20 |
| 2 | −150 | +50 | Max(50×10%−20, 0) = 0 | 0 | 20 | 水位线以下，无分润 |
| 3 | +100 | +150 | Max(150×10%−20, 0) = 0 | 0 | 20 | 15−20<0，仍在水位线下 |
| 4 | +150 | +300 | Max(300×10%−20, 0) = 10 | 10 | 20 | 超过水位线，新增分润 10 |
| 结算 | — | — | 0 | 0 | 30 | 支付 10，settled 累计 30 |

**关键点**：第 2、3 周即使有盈利，但累计利润对应的分润总额（5、15）低于已结算金额（20），HWM 生效，无新增分润。

#### Case 2：有仓位直接结算 — Full MTM

| 事件 | 已实现净利润 | 未实现PnL | 应计基准 | accrued | reserved | settled | status |
|------|------------|----------|---------|---------|----------|---------|--------|
| 周一窗口（有持仓） | +500 | −100 | 500+(−100)=400 | 40 | 40 | 0 | **payable_pending** |
| 执行支付 | — | — | — | 0 | 0 | 40 | **settled** → normal |
| 周三全平 | +400（含平仓亏损） | 0 | 400+0=400 | Max(400×10%−40,0)=0 | 0 | 40 | normal |

**关键点**：
- Full MTM 下有仓位时正常结算支付，不再递延
- 周一浮亏 −100 自动纳入计算：基准=400，accrued=40（不是 50）
- 周三全平后已实现=+400，accrued=Max(40−40,0)=0，无新增分润（已在周一结算）

#### Case 3：持续持仓 — 每周正常结算（Full MTM）

| 事件 | 已实现净利润 | 未实现PnL | 应计基准 | accrued | reserved | settled | status |
|------|------------|----------|---------|---------|----------|---------|--------|
| 第 1 周窗口 | +300 | +50 | 300+50=350 | 35 | 35 | 0 | payable_pending |
| 支付 | — | — | — | 0 | 0 | 35 | settled → normal |
| 第 2 周窗口 | +300 | −80 | 300+(−80)=220 | Max(22−35,0)=0 | 0 | 35 | normal（HWM 以下） |
| 第 3 周窗口 | +400 | +100 | 400+100=500 | Max(50−35,0)=15 | 15 | 35 | payable_pending |
| 支付 | — | — | — | 0 | 0 | 50 | settled → normal |

**关键点**：
- Full MTM 下每周都正常结算，浮盈浮亏均参与计算
- 第 1 周浮盈 +50 参与计算，直接支付 35
- 第 2 周浮亏扩大，基准降至 220，分润 22 < 已结算 35，HWM 生效，无新增分润
- 第 3 周已实现增加且浮盈恢复，超过 HWM 水位线，新增分润 15

#### Case 4：insufficient 状态 — 跟单者余额不足

> 假设 reserve 在上一次窗口刷新时正常确认为 50（当时可分配余额 ≥ 50），但刷新后到本次支付前，因市场波动或费用扣减导致可分配余额从 ≥50 降至 30。

| 事件 | accrued | reserved | 可分配余额 | status |
|------|---------|----------|---------|--------|
| 周一窗口，无仓位 | 50 | 50 | 30 | — |
| 尝试支付 | — | — | 30 < 50 | **insufficient** |
| 跟单者追加入金 60 | 50 | 50 | 90 | — |
| 下一窗口重试 | 50 | 50 | 90 ≥ 50 | **payable_pending** → settled |

**关键点**：余额不足时不减免，挂起等待。跟单者追加入金后下一窗口重试。

#### Case 5：一周内多次清仓

> 假设各结算时点均已全平，未实现 PnL = 0，统一公式退化为纯已实现口径。

| 事件 | 已实现净利润 | accrued | settled |
|------|------------|---------|---------|
| 周一全平 | +200 | 20 | 0 |
| 支付 | — | 0 | 20 |
| 周三重新开仓并全平 | +350 | Max(350×10%−20,0) = 15 | 20 |
| 支付 | — | 0 | 35 |

**关键点**：HWM 是连续的。第二次结算时 settled 已包含第一次的 20，所以新增分润只有 15（不是 35×10%=35）。

### 4.13 边界案例表

| 场景 | 处理规则 |
|------|---------|
| 带单员退出（停止带单/注销） | 触发即时结算：仓位全平 → 重算 accrued → reserved → 执行支付 |
| 平台封禁带单员 | 仓位强平 → 重算 accrued → reserved **冻结**，不立即支付，待调查结果决定是否发放 |
| 跟单者主动停止跟单 | 仓位全平 → 触发即时结算 → 正常支付 |
| 系统自动取消跟随（本金不足/30日无开仓） | 仓位跟随带单员后续平仓 → 全部平仓后触发结算 |
| 平台/客服人工终止特定跟单关系 | 属于平台干预场景，非带单员产品功能。终止后按"关系终止"标准流程结算：仓位全平 → 重算 accrued → reserved → 执行支付。具体终止条件与结算细节由平台清结算与风控规则决定 |
| 一周内多次全平再开仓 | HWM 连续，多次结算各自独立但 settled 累计（见 Case 5） |
| 跟单者追加入金 | 不影响已实现净利润计算，不影响 HWM |
| 带单员等级升级 | 已建立的跟单关系按原比例，新建立的跟单关系按新比例 |
| 强平导致跟单仓位全平 | 触发即时结算；强平产生的亏损纳入已实现净利润 |
| 账户级止损触发 | 仓位全平，停止跟单 → 触发即时结算 |
| 带单员故意留小仓位（规避结算） | Full MTM 下每周窗口正常结算，无需平仓即可支付，留小仓位不影响结算节奏 |
| 跟单者余额不足以覆盖 reserved | 进入 insufficient 状态，不减免，等待余额恢复（见 Case 4） |
| 关系终止时跟单者余额仍不足 | 从最终可分配余额中尽可能支付；不足部分进入未结清状态，后续处理由平台清结算与风控规则决定 |
| 锁定期内到达结算窗口 | 正常执行结算流程（锁定期限制的是资金操作和停止跟单，不限制分润结算） |
| 分润比例为 0 | 不会发生（规则要求 >0），如数据异常则 accrued = 0 |
| 累计已实现净利润 ≤ 0 | accrued = 0，不产生任何分润 |

### 4.14 带单员退出与封禁时的分润处理

#### 4.14.1 主动退出（停止带单/注销）

1. 系统立即将所有跟单者仓位**市价全平**
2. 全平产生的盈亏**纳入正常已实现净利润**
3. 触发即时结算，按标准流程计算 accrued → reserved → settled
4. 退出原因不影响分润结算，已发生的交易事实不受影响

#### 4.14.2 平台封禁

1. 立即冻结带单功能
2. 所有关联跟单者仓位**立即市价强制全平**
3. 全平产生的盈亏纳入已实现净利润
4. 计算 accrued 和 reserved，`settlement_status` 进入 `payable_pending`，同时设置 `reserve_hold_reason = lead_banned`
5. 由于 `reserve_hold_reason` 非空，系统**阻止实际支付**，reserved 进入冻结状态
6. 待调查结果：
   - 确认无违规 → 将 `reserve_hold_reason` 置为 `null`，解冻后正常支付
   - 确认违规 → reserved 不发放，按平台规则处理

#### 4.15.1 跟单者端

| 展示项 | 数据来源 |
|--------|---------|

| 待结算分润 | `reserved_profit_share`（单独一行展示） |

#### 4.15.2 带单员端

| 当前待结算分润 | 所有活跃关系中 `reserve_hold_reason = null` 的 `reserved_profit_share` 之和（可正常结算） |
| 当前冻结待结算分润 | 所有活跃关系中 `reserve_hold_reason ≠ null` 的 `reserved_profit_share` 之和（调查中/暂不可支付） |
| 预估应计分润 | 所有活跃关系的 `accrued_profit_share` 之和（实时参考值） |

### 4.16 金额精度与舍入规则

| 项目 | 规则 |
|------|------|
| **稳定币精度** | USDC/USDT 均保留 **2 位小数** |
| **分润计算中间步骤** | 至少保留 **6 位小数**，避免累积舍入误差 |
| **最终支付金额** | 向下取整至 2 位小数（truncate，非四舍五入），保护跟单者 |
| **最小支付阈值** | 低于 **0.01 USDC** 的 `reserved_profit_share` 不触发支付，累积到下次结算 |

## 五、限制与风控

### 5.1 带单员风控告警

以下情况各触发 1 次风控告警：

1. 带单员同时**双向开仓**超过 **5 笔**
2. 带单员**带单仓位亏损累计时间**超过 **24 小时**未平仓
3. 带单员存在**多笔或多账号开仓并平仓时间趋于相同**（疑似操控）
4. 带单员近 **7 日无新增带单交易**

> 风控告警仅用于监控和人工介入判断，不直接等同于自动封禁或自动终止带单，除非平台另有明确风控规则。

### 5.2 跟单交易限制

1. **体验金**不可用于抵消跟单交易的保证金与交易费用
2. 跟单者因**本金不足**跟单失败 **1 次**，系统**立即自动取消**跟随该带单员

### 5.3 跟单者数量

带单员的跟随者**不设数量上限**。

当某次跟单中，**带单金额总计（带单员 + 所有跟单者）超过平台该币种可开上限**时：
- 带单员优先占用额度
- 剩余额度按各跟单者金额占比 pro-rata 分配
- 分配后若某跟单者的实际可开金额 < 最小开仓量，该笔跟单跳过

## 六、开平仓规则

1. 带单员通过**专用带单路由**在交易页开仓，**限价、市价、计划单均支持**
2. 跟单者根据带单员**开仓成交订单**自动跟单；订单拆分规则对齐币安：**市价单**立即触发跟单（IOC 限价单模式），**限价单**完全成交后触发跟单（跟单者以市价执行），**多笔独立订单**各自单独触发
3. 带单员平仓后，当前仓位关联的所有跟单者仓位**全部平仓**；支持市价平仓、市价全平、止盈止损触发平仓
4. 带单员带单仓位被**强平**时，所有关联跟单者仓位同步**触及平仓**
5. 跟单者**主动停止跟单**后，所有跟单仓位**立即市价全平**，剩余资金转回

## 七、异常与边界

| 场景 | 处理 |
|------|------|
| 申请时仍有仓位/委托/计划单 | 申请失败 |
| 申请时仍有跟随者 | 申请失败 |
| 人工审核不通过 | 申请失败 |
| 跟单者资产不足最低跟单金额 | 无法跟随该带单员 |
| 带单员近 30 日无开仓记录 | 无法发起新跟随；已有跟随者系统自动取消跟随 |
| 跟单者单次本金不足 | 本次跟单失败，系统**立即**自动取消跟随 |
| 带单金额总计超平台该币种上限 | 带单员优先，剩余额度按跟单者占比 pro-rata 分配；分配后低于最小开仓量则跳过 |
| 带单员申请移除特定跟随者 | 带单员无权单方面移除或终止特定跟随关系；如需处理个别跟单者，仅可通过客诉/申诉/运营渠道申请，由平台人工处理 |
| 邀请码被 disable | 该码不可再被新跟随者使用；已通过该码加入的跟随者不受影响 |
| 跟单者主动停止跟单 | 所有跟单仓位立即市价全平，资金转回 |
| 带单员停止带单/注销 | 所有跟单者仓位立即市价全平；触发即时分润结算（见 4.14.1 节） |
| 平台封禁带单员 | 立即强平所有跟单仓位；计算 accrued/reserved 后冻结 reserved（详见 4.14.2 节）；站内信通知跟单者 |
| 带单员带单仓位强平 | 所有关联跟单仓位同步平仓 |
| 账户级止损触发 | 所有跟单仓位市价全平，停止跟单，资金转回 |
| 锁定期内请求停止跟单或资金操作 | 操作被阻止；锁定期结束前不可执行 |
| 跟单进行中请求转出资金 | 操作被阻止；需先停止跟单（仓位全平）后方可转出 |
| 订单拆分与跟单触发 | 市价单立即触发（IOC 限价单模式）；限价单完全成交后触发（跟单者市价执行）；多笔独立订单各自单独触发 |
| 累计已实现净利润 ≤ 0 | accrued = 0，不产生分润（HWM 机制，见 4.4 节） |
